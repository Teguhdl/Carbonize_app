ABSTRAK

Perubahan iklim global yang dipicu oleh emisi
karbon dioksida dari aktivitas antropogenik menuntut adanya upaya peningkatan
kesadaran publik terhadap jejak karbon individual. Aplikasi Carbonize
dikembangkan sebagai platform seluler untuk membantu pengguna mengukur jejak
karbon dari aktivitas sehari-hari seperti penggunaan transportasi dan konsumsi
produk sekali pakai. Namun, evaluasi terhadap versi awal aplikasi
mengidentifikasi beberapa masalah teknis yang signifikan, meliputi struktur
kode yang kurang terorganisir dengan adanya duplikasi logika, kurangnya
modularisasi, tidak adanya pemisahan jelas antara lapisan logika bisnis dan
antarmuka pengguna, serta ketergantungan pada Firebase yang memiliki
keterbatasan dalam manajemen data terstruktur. Penelitian ini bertujuan untuk
melakukan refactoring dan perbaikan aplikasi Carbonize dengan fokus pada
peningkatan struktur kode dan migrasi sistem basis data dari Firebase ke
Database Management System (DBMS) relasional seperti MySQL atau PostgreSQL.
Metode yang digunakan meliputi penerapan prinsip modularitas, pemisahan
kepentingan (separation of concerns), dan praktik terbaik dalam pengembangan
perangkat lunak, serta perancangan dan implementasi skema basis data relasional
yang lebih terstruktur. Evaluasi dampak refactoring dan migrasi dilakukan
menggunakan metrik kualitas kode seperti cyclomatic complexity, coupling, dan
cohesion untuk aspek maintainability, serta response time dan query execution
time untuk aspek performa sistem. Hasil penelitian diharapkan menghasilkan
arsitektur perangkat lunak yang lebih teratur, mudah dipelihara, dan dapat
diandalkan tanpa mengubah fungsionalitas utama aplikasi. Penelitian ini
memberikan kontribusi teoritis berupa rujukan implementasi clean code dan
modularitas pada aplikasi mobile, serta manfaat praktis berupa peningkatan
stabilitas sistem, kemudahan pengembangan fitur masa depan, dan jaminan
pengelolaan data pengguna yang lebih baik. Batasan penelitian meliputi fokus
pada aspek teknis tanpa penambahan fitur baru, platform terbatas pada aplikasi
mobile, dan pengujian dalam skala lingkungan pengembangan. Penelitian ini
menyediakan fondasi yang kokoh untuk pengembangan lebih lanjut aplikasi
Carbonize dan dapat menjadi panduan bagi aplikasi serupa dalam melakukan
refactoring dan migrasi basis data.

Kata Kunci: refactoring, migrasi basis data,
aplikasi mobile, jejak karbon, maintainability, DBMS relasional, Firebase,
kualitas kode.

 

ABSTRACT

Global climate
change driven by carbon dioxide emissions from anthropogenic activities
necessitates efforts to increase public awareness of individual carbon
footprints. The Carbonize application was developed as a mobile platform to
help users measure carbon footprints from daily activities such as
transportation use and consumption of disposable products. However, evaluation
of the initial version of the application identified several significant
technical issues, including poorly organized code structure with logic
duplication, lack of modularization, absence of clear separation between
business logic and user interface layers, and dependency on Firebase which has
limitations in structured data management. This research aims to refactor and
improve the Carbonize application with a focus on enhancing code structure and
migrating the database system from Firebase to relational Database Management
Systems (DBMS) such as MySQL or PostgreSQL. The methods employed include
applying modularity principles, separation of concerns, and software
development best practices, as well as designing and implementing a more
structured relational database schema. Evaluation of the impact of refactoring
and migration is conducted using code quality metrics such as cyclomatic
complexity, coupling, and cohesion for maintainability aspects, as well as
response time and query execution time for system performance aspects. The
research results are expected to produce a more organized, maintainable, and
reliable software architecture without altering the core functionality of the
application. This research provides theoretical contributions in the form of
reference implementation of clean code and modularity in mobile applications,
as well as practical benefits including improved system stability, ease of
future feature development, and better assurance of user data management.
Research limitations include a focus on technical aspects without adding new
features, platform limited to mobile applications, and testing in development
environment scale. This research provides a solid foundation for further
development of the Carbonize application and can serve as a guide for similar
applications in conducting refactoring and database migration.

Keywords: refactoring, database migration, mobile
application, carbon footprint, maintainability, relational DBMS, Firebase, code
quality.

BAB I PENDAHULUAN

I.1.         
Latar Belakang

Perubahan iklim global tetap menjadi tantangan yang signifikan di abad
ini, dengan eskalasi emisi karbon dioksida yang dihasilkan dari aktivitas
antropogenik yang diidentifikasi sebagai faktor utama. Seperti yang digambarkan
dalam laporan oleh Panel Antarpemerintah tentang Perubahan Iklim (IPCC, 2021), upaya manusia telah memicu kenaikan suhu global sekitar 1,1° C relatif
terhadap zaman pra-industri, dengan emisi karbon dioksida mencapai tingkat yang
belum pernah terjadi sebelumnya dalam sejarah kontemporer. Untuk meningkatkan
kesadaran publik mengenai konsekuensi kegiatan sehari-hari terhadap lingkungan,
aplikasi Carbonize telah disusun, yang merupakan platform seluler yang
direkayasa untuk membantu pengguna dalam mengukur jejak karbon yang terkait
dengan perilaku seperti penggunaan transportasi dan konsumsi produk sekali
pakai. Aplikasi ini bertujuan untuk memberi pengguna wawasan tentang kontribusi
individu mereka terhadap emisi karbon, sementara secara bersamaan menumbuhkan
kesadaran akan pentingnya pengurangan emisi dalam kehidupan sehari-hari.

Namun, berdasarkan hasil evaluasi terhadap versi awal aplikasi Carbonize
yang sebelumnya dikembangkan, teridentifikasi beberapa masalah yang berfokus
pada aspek teknis sistem. Salah satu kendala teknis utama yang teridentifikasi
adalah struktur kode yang bersifat monolitik dan tidak terorganisir, yang
ditandai dengan munculnya fenomena God Class pada berkas utama (seperti
calculator_screen.dart) yang mencapai lebih dari 4.400 baris kode dalam satu
unit tunggal. Beberapa bagian dari kode menunjukkan adanya duplikasi logika,
kurangnya pembagian modul, serta tidak adanya pemisahan yang jelas antara
lapisan logika bisnis dan antarmuka pengguna. Situasi ini menyebabkan aplikasi
lebih sulit dipelihara, meningkatkan tingkat kompleksitas dalam pengembangan
lanjutan, serta meningkatkan risiko munculnya kesalahan (bug) ketika terjadi
perubahan atau penambahan kode. Menurut penelitian (Sharma & Mishra, 2012),
masalah seperti duplikasi kode dan kurangnya modularisasi dapat mengurangi
kemudahan pemeliharaan hingga 40% dan secara signifikan meningkatkan teknis
debt.

Selain tantangan pada struktur kode, aplikasi Carbonize juga masih
bergantung pada Firebase sebagai platform utama untuk penyimpanan data.
Walaupun Firebase memudahkan proses integrasi dan mempercepat pengembangan,
sistem ini memiliki keterbatasan dalam manajemen data yang terstruktur,
terutama jika dibandingkan dengan Sistem Manajemen Basis Data (DBMS)
tradisional. Keterbatasan tersebut mencakup dukungan yang kurang optimal untuk
relasi antar data, pengaturan transaksi, serta keleluasaan dalam pemrosesan dan
analisis data yang rumit (Khawas & Shah, 2018). Hal ini menciptakan
kesenjangan antara kebutuhan sistem yang ideal yang mengharuskan pengelolaan
data yang teratur dan terstruktur dengan kondisi implementasi saat ini yang
masih tergolong sederhana.

Berdasarkan isu yang ada, studi ini ditujukan untuk melakukan refactoring
dan perbaikan pada aplikasi Carbonize, dengan fokus pada aspek teknis, seperti
peningkatan struktur kode dan penggantian sistem basis data. Refactoring
dilakukan dengan merombak kode agar menjadi lebih modular, terstruktur, dan
lebih mudah untuk dipelihara tanpa mengubah fungsionalitas utama aplikasi.
Metode ini sejalan dengan prinsip refactoring yang bertujuan untuk memperbaiki
kualitas desain internal perangkat lunak tanpa mempengaruhi perilaku eksternal
sistem (Fowler, 2018) .

Selain itu, dilakukan perpindahan sistem penyimpanan informasi dari
Firebase ke DBMS seperti MySQL atau PostgreSQL. Penggunaan DBMS diharapkan
dapat memberikan pengelolaan data yang lebih sistematis, konsisten, dan
efisien, sehingga mendukung kestabilan sistem dan memudahkan pengelolaan data
dalam jangka panjang. Penelitian oleh (Gy&#337;rödi, Gy&#337;rödi, & Sotoc,
2015) mengungkapkan bahwa DBMS relasional seperti MySQL dan PostgreSQL memiliki
kelebihan dalam menangani operasi kompleks, menjaga integritas data, serta
skalabilitas untuk aplikasi yang membutuhkan struktur data yang terorganisir.

Dengan melakukan perbaikan pada struktur kode serta mengganti sistem
penyimpanan data, diharapkan aplikasi Carbonize akan memiliki arsitektur
perangkat lunak yang lebih teratur, lebih mudah dalam pemeliharaan, dan lebih
dapat diandalkan. Fokus dari perbaikan ini bukanlah pada penambahan fitur-fitur
baru, melainkan pada peningkatan kualitas internal sistem sehingga aplikasi
dapat terus dikembangkan dan siap memenuhi kebutuhan teknis di masa yang akan
datang.

Rumusan Masalah

Berdasarkan latar belakang di atas, maka dirumuskan beberapa permasalahan
sebagai berikut:

 Bagaimana cara
     merombak struktur kode aplikasi Carbonize agar lebih modular dan

 Bagaimana proses migrasi penyimpanan data

 Bagaimana peningkatan kualitas maintainability
     (kemudahan pemeliharaan) kode setelah dilakukan proses refactoring?

I.3.         
Tujuan Penelitian

Berdasarkan rumusan masalah yang telah disusun, tujuan penelitian ini
adalah:

1.    
Menerapkan prinsip
modularitas dan pemisahan kepentingan pada struktur kode aplikasi Carbonize agar menjadi lebih teratur dan
memenuhi standar penulisan kode yang baik (best practices).

2.    
Merancang dan
mengimplementasikan migrasi sistem penyimpanan data dari Firebase ke Sistem Manajemen Basis Data (DBMS)
relasional guna meningkatkan keteraturan dan konsistensi pengelolaan data.

3.    
Mengevaluasi
peningkatan kualitas maintainability (kemudahan pemeliharaan) pada aplikasi Carbonize
sebelum dan sesudah proses refactoring dengan menggunakan metrik
perangkat lunak yang terukur.

acuan sistem.

II.1.1.  Definisi dan Konsep Jejak Karbon

Jejak karbon (carbon footprint) dapat didefinisikan sebagai total emisi gas
rumah kaca (GRK) yang dihasilkan dari berbagai aktivitas individu, organisasi,
atau produk dalam suatu periode waktu tertentu. Emisi ini meliputi gas-gas
seperti karbon dioksida (CO&#8322;), metana (CH&#8324;), dan dinitrogen oksida
(N&#8322;O), yang semuanya berkontribusi terhadap fenomena pemanasan global dan
perubahan iklim (Greenia, 2025). Jejak karbon juga dapat dipahami sebagai total
emisi gas rumah kaca yang ditimbulkan oleh aktivitas manusia, baik secara
langsung maupun tidak langsung. Definisi ini menekankan bahwa semua bentuk
kegiatan manusia memiliki implikasi terhadap pelepasan gas-gas yang memicu efek
rumah kaca ke atmosfer bumi. Pengukuran jejak karbon mencakup volume gas rumah
kaca yang dilepaskan ke atmosfer sebagai konsekuensi dari beragam aktivitas
manusia, seperti penggunaan kendaraan bermotor, konsumsi listrik, dan proses produksi
barang. Oleh karena itu, besaran jejak karbon setiap individu dapat bervariasi
secara signifikan, tergantung pada gaya hidup, lokasi geografis, dan jenis
kegiatan yang dilakukan (Arumka, 2025).

Seiring meningkatnya
kesadaran akan krisis iklim global, penggunaan teknologi telah menjadi strategi
krusial untuk mengedukasi dan mendorong perubahan perilaku yang lebih
berkelanjutan. Dalam konteks ini, pengembangan aplikasi seluler untuk
perhitungan dan reduksi jejak karbon menjadi pendekatan inovatif yang sangat
menjanjikan. Aplikasi semacam ini membekali pengguna dengan kapabilitas untuk
memantau, menganalisis, dan secara proaktif mengurangi emisi karbon yang
dihasilkan dari rutinitas harian mereka, mulai dari kebiasaan transportasi,
konsumsi energi di rumah, hingga pola konsumsi barang dan jasa (Ojadi, Owulade, Odionu, & Onukwulu, 2025). Kemampuan aplikasi untuk memberikan umpan balik
real-time dan rekomendasi personal telah terbukti efektif dalam meningkatkan
kesadaran lingkungan dan memotivasi pengguna untuk mengadopsi gaya hidup yang
lebih berkelanjutan (Roxanne, Rasyidin, & Setijadi, 2023)

II.1.2.    Perhitungan Jejak Karbon

Menurut (IPB University, 2019), Perhitungan jejak karbon (carbon footprint) merupakan proses
kuantifikasi total emisi gas rumah kaca (GRK), terutama karbon dioksida
ekuivalen (CO&#8322;e), yang dihasilkan secara langsung maupun tidak langsung
dari aktivitas manusia. Emisi langsung mencakup kegiatan seperti penggunaan
kendaraan bermotor berbahan bakar fosil, sementara emisi tidak langsung dapat
berasal dari konsumsi energi listrik yang diproduksi dari sumber energi
non-terbarukan. jejak karbon mengukur kontribusi individu maupun organisasi
terhadap perubahan iklim melalui emisi GRK yang ditimbulkan dari aktivitas
sehari-hari.

Secara umum, proses
perhitungan jejak karbon mencakup beberapa aspek utama, antara lain
transportasi, konsumsi energi rumah tangga, dan gaya hidup. Dalam aspek
transportasi, perlu diperhitungkan jenis kendaraan yang digunakan, total jarak
tempuh, serta efisiensi bahan bakar untuk mengestimasi emisi yang dihasilkan.
Perjalanan udara juga memberikan kontribusi signifikan terhadap jejak karbon
dan harus dimasukkan dalam perhitungan. Sementara itu, dalam aspek konsumsi
energi rumah tangga, analisis difokuskan pada penggunaan listrik dan gas,
dengan mempertimbangkan sumber energi lokal, serta penggunaan peralatan
elektronik, sistem pencahayaan, pemanas, dan pendingin ruangan. Langkah-langkah
mitigasi seperti penggunaan energi terbarukan dan penerapan gaya hidup hemat
energi dapat secara signifikan mengurangi besarnya jejak karbon (Abdullah,
2024).

Perhitungan jejak
karbon umumnya mengacu pada metode yang disarankan oleh (Intergovernmental
Panel on Climate Change, 2006) dalam dokumen 2006 Guidelines for National
Greenhouse Gas Inventories. Metode tersebut bersifat kuantitatif dan
menggunakan formula standar:

merujuk pada besaran atau volume aktivitas yang menyebabkan emisi, misalnya

yang ditempuh.

·      
Faktor Emisi adalah
nilai tetap yang mewakili jumlah CO&#8322;e yang dilepaskan per unit aktivitas.
Nilai ini ditetapkan oleh lembaga seperti IPCC atau negara melalui pedoman
nasional (misalnya, 2.31 kg CO&#8322; per liter bensin).

II.1.3.  Ruang Lingkup Emisi Karbon

Setelah memahami
berbagai jenis Gas Rumah Kaca (GRK) dan cara membandingkan potensinya melalui
GWP, langkah selanjutnya dalam inventarisasi emisi adalah mengkategorikan
sumber-sumber emisi tersebut. Untuk tujuan standardisasi dan konsistensi
pelaporan, kerangka kerja yang paling diakui secara global adalah The
Greenhouse Gas Protocol: A Corporate Accounting and Reporting Standard
(selanjutnya disebut GHG Protocol) (The Greenhouse Gas Protocol, 2004) .
Kerangka kerja ini memperkenalkan konsep &quot;Ruang Lingkup&quot; atau Scopes
untuk membedakan antara emisi yang dihasilkan secara langsung oleh sebuah
entitas dengan emisi tidak langsung yang terjadi akibat aktivitas entitas
tersebut.

Pengkategorian ini
krusial untuk menghindari penghitungan ganda antar entitas dan untuk memberikan
gambaran yang transparan mengenai carbon footprint. GHG Protocol membagi emisi
ke dalam tiga lingkup utama, yaitu:

1.     Scope 1: Emisi Langsung, Kategori ini mencakup seluruh emisi GRK yang
berasal dari sumber-sumber yang dimiliki atau dikendalikan secara langsung oleh
sebuah entitas (The Greenhouse Gas Protocol, 2004). Contohnya termasuk emisi
dari proses pembakaran pada mesin atau kendaraan yang dimiliki, emisi dari
proses produksi kimia, dan emisi fugitif (kebocoran). Dalam konteks rumah
tangga dan aplikasi pengukuran jejak karbon aktivitas domestik, emisi yang dihasilkan
dari penggunaan gas LPG untuk memasak termasuk dalam kategori Scope 1, karena
pengguna memiliki kendali langsung atas sumber emisi tersebut melalui
penggunaan kompor gas di rumah.

2.    
Scope 2: Emisi Tidak Langsung dari Energi, Kategori
ini mencakup emisi GRK tidak langsung yang berasal dari produksi energi yang
dibeli dan dikonsumsi oleh sebuah entitas, seperti listrik, uap, panas, atau
pendingin (The Greenhouse Gas Protocol, 2004). Meskipun entitas tersebut
mengonsumsi energi, emisi fisiknya terjadi di lokasi pembangkit listrik atau
fasilitas produksi energi lainnya yang tidak mereka miliki atau kendalikan (The Greenhouse Gas Protocol, 2004). Dalam konteks aktivitas rumah tangga, emisi dari
penggunaan peralatan elektronik seperti televisi, kulkas, AC, rice cooker, dan
perangkat lainnya yang menggunakan listrik PLN merupakan contoh utama dari
Scope 2, karena emisi fisik terjadi di pembangkit listrik namun konsumsi energi
dilakukan di rumah tangga.

3.     Scope 3: Emisi Tidak Langsung Lainnya, Ini adalah kategori yang paling
luas, mencakup semua emisi tidak langsung lainnya (yang tidak termasuk dalam
Scope 2) yang terjadi di sepanjang rantai nilai sebuah entitas, baik di sisi
hulu (upstream) maupun hilir (downstream) (The Greenhouse Gas
Protocol, 2004). Ini termasuk emisi dari ekstraksi dan produksi material atau
bahan baku yang dibeli, transportasi yang tidak dimiliki atau dikendalikan
perusahaan, hingga penggunaan dan pembuangan produk yang dijual (The
Greenhouse Gas Protocol, 2021). Dalam konteks rumah tangga, aktivitas seperti
konsumsi bahan makanan (daging, sayuran, beras) yang proses produksi dan
distribusinya menghasilkan emisi, serta penggunaan peralatan elektronik yang
mempertimbangkan emisi dari proses manufaktur dan pembuangan akhir perangkat
tersebut, termasuk dalam kategori Scope 3.

Untuk memberikan
panduan yang lebih akurat dalam perhitungan emisi karbon, Intergovernmental
Panel on Climate Change (IPCC) telah menerbitkan penyempurnaan terbaru dari
panduan inventarisasi GRK pada tahun 2019. Dokumen &quot;2019 Refinement to the
2006 IPCC Guidelines for National Greenhouse Gas Inventories&quot; diadopsi dan
diterima selama Sesi ke-49 IPCC pada Mei 2019 (IPCC, 2019). Penyempurnaan ini
memberikan metodologi tambahan untuk memperkirakan sumber-sumber yang
menghasilkan emisi gas rumah kaca dan penyerap yang menyerap gas-gas tersebut.

Penyempurnaan IPCC
2019 ini menyediakan metodologi yang telah diperbarui, faktor emisi default,
dan parameter lainnya berdasarkan informasi ilmiah terbaru untuk sumber dan
penyerap GRK (Greenhouse Gas Management Institute (GHGMI), 2020). Salah satu
pembaruan penting adalah panduan baru tentang emisi GRK tidak langsung dari
sumber non-biogenik CO&#8322; dari oksidasi atmosferik metana (CH&#8324;),
karbon monoksida (CO), dan senyawa organik volatil non-metana (NMVOCs) (Greenhouse Gas Management Institute (GHGMI), 2020).

Selain
mengkategorikan sumber emisi melalui Scopes, IPCC juga menyediakan pendekatan
berjenjang atau Tiers untuk menentukan metodologi perhitungan emisi karbon
dengan tingkat kerincian yang berbeda (IPCC, 2019). Tier ini merepresentasikan
tingkat kerincian metode dan kualitas data yang digunakan, dengan tingkat
kompleksitas yang meningkat dari Tier 1 ke Tier 3. Tier 1 adalah metode yang
paling dasar, Tier 2 menengah, dan Tier 3 paling menuntut dalam hal kompleksitas
dan persyaratan data (Snapshot Climate, n.d.). Tier 2 dan 3 kadang disebut
sebagai metode &quot;tier yang lebih tinggi&quot; dan umumnya dianggap lebih
akurat serta lebih cocok untuk tindakan mitigasi MRV (Monitoring, Reporting,
and Verification).

1.     Tier 1: Menggunakan faktor emisi default yang disediakan IPCC dengan
data aktivitas yang sederhana.

2.    
Tier 2: Menggunakan faktor emisi yang lebih
spesifik berdasarkan kondisi negara atau wilayah. 

3.     Tier 3: Menggunakan metode yang lebih detail dengan model atau sistem
pengukuran yang kompleks.

Dengan demikian,
aplikasi mobile untuk penghitungan jejak karbon aktivitas rumah tangga secara
praktis membantu pengguna untuk melacak emisi dari Scope 1 (penggunaan gas LPG
untuk memasak), Scope 2 (konsumsi listrik dari peralatan elektronik), dan
sebagian dari Scope 3 (konsumsi bahan makanan dan siklus hidup perangkat
elektronik). Mengacu pada penyempurnaan IPCC 2019, aplikasi dapat
mengintegrasikan faktor emisi yang telah diperbarui dan metodologi yang lebih
akurat untuk perhitungan jejak karbon dari aktivitas penggunaan energi dan penggunaan
energi untuk keperluan dapur di rumah tangga. Pendekatan Tier yang sesuai dapat
dipilih berdasarkan ketersediaan data dan tingkat akurasi yang diinginkan,
dengan Tier 1 sebagai pendekatan dasar yang dapat diimplementasikan untuk
pengguna umum dalam menghitung emisi dari penggunaan peralatan elektronik dan
kegiatan memasak sehari-hari.

II.2.       Pengembangan Aplikasi Mobile Android

Setelah memahami
pentingnya perhitungan jejak karbon serta ruang lingkup emisi yang relevan
dengan aktivitas rumah tangga, langkah selanjutnya adalah membahas bagaimana
teknologi, khususnya aplikasi mobile, dapat dimanfaatkan sebagai alat bantu
praktis dalam proses pelacakan dan mitigasi emisi karbon. Mengingat penetrasi
perangkat seluler yang tinggi di masyarakat serta kemampuannya untuk memberikan
umpan balik secara real-time, aplikasi mobile menjadi sarana yang sangat
efektif dalam mendorong perilaku berkelanjutan berbasis data.

Menurut (Pressman & Maxim, Software Engineering : A Practitioners Approach, 2019), aplikasi
mobile adalah perangkat lunak yang dirancang secara khusus untuk dijalankan
pada sistem operasi perangkat seluler seperti iOS, Android, atau Windows
Mobile. Aplikasi ini dikembangkan dengan mempertimbangkan karakteristik dan
keterbatasan perangkat seluler, seperti ukuran layar, daya komputasi, serta
interaksi berbasis sentuhan. Antarmuka pengguna (user interface)
aplikasi mobile dirancang agar sesuai dengan mekanisme interaksi khas
dari masing-masing platform, seperti gesture, swipe, dan tap.
Selain itu, aplikasi mobile dapat terhubung dengan sumber daya daring (web-based
resources) untuk mengakses informasi eksternal yang relevan, serta memiliki
kemampuan pemrosesan data secara lokal. Hal ini memungkinkan aplikasi untuk
mengumpulkan, menganalisis, dan menyajikan informasi secara efisien, sekaligus
menyediakan penyimpanan data yang bersifat persisten di dalam perangkat
pengguna.

Dalam konteks
penerapan aplikasi mobile untuk isu lingkungan, penelitian oleh (Hasan, Prijanto, & Setyoko, 2023) mengembangkan sebuah prototipe aplikasi bernama Carbon
Footprint Survey. Aplikasi ini dirancang untuk mencatat konsumsi energi
rumah tangga, termasuk penggunaan listrik, gas, dan bahan bakar, serta
menghitung total emisi karbon dioksida ekuivalen (CO&#8322;e) berdasarkan
standar perhitungan yang berlaku. Aplikasi tersebut diterapkan di wilayah kerja
puskesmas, dengan tujuan tidak hanya sebagai alat pemantau emisi rumah tangga
secara akurat, tetapi juga sebagai media edukasi bagi masyarakat untuk
meningkatkan kesadaran dalam menjaga dan mengurangi emisi individu.

Beberapa perusahaan
teknologi di Indonesia telah mulai mengintegrasikan fitur jejak karbon dalam
aplikasi mereka untuk mendorong perubahan perilaku pengguna secara sukarela.
Dua contoh aplikasi yang telah diterapkan secara nyata adalah:

·      
GoGreener oleh Gojek

GoGreener
merupakan fitur dalam aplikasi Gojek yang memungkinkan pengguna menghitung
jejak karbon dari penggunaan layanan transportasi mereka, serta melakukan
kompensasi dengan menanam pohon. Gojek menyatakan bahwa fitur ini dikembangkan
bekerja sama dengan Jejak.in, sebuah startup yang fokus pada teknologi
lingkungan. Dalam fitur ini, pengguna dapat melihat estimasi emisi CO&#8322;
yang dihasilkan dari perjalanan mereka dan memilih untuk menanam pohon secara
digital untuk mengimbanginya (Gojek, 2024).

 

·      
Carbon Offset oleh MyTelkomsel

MyTelkomsel
meluncurkan fitur &quot;Carbon Offset Program&quot; bekerja sama dengan
Jejak.in, di mana pengguna dapat menukarkan Telkomsel Poin untuk berkontribusi
dalam penanaman pohon dan pengimbangan emisi karbon. Setiap penukaran poin
disetarakan dengan sejumlah kilogram emisi CO&#8322; yang dikompensasi. Program
ini ditujukan untuk meningkatkan kesadaran pelanggan terhadap perubahan iklim
serta memberikan opsi nyata dalam mengurangi dampak lingkungan (Telkomsel, 2022).

Namun, selain
contoh-contoh lokal tersebut, terdapat pula berbagai aplikasi independen global
yang telah dirancang secara khusus untuk menghitung jejak karbon individu
dengan cakupan aktivitas yang luas dan fitur yang lebih personal. Penambahan
ini penting sebagai referensi dalam merancang aplikasi penelitian ini agar
dapat mengadopsi praktik terbaik yang sudah terbukti:

I.     
EcoTrack 

Aplikasi
ini mengintegrasikan kalkulator karbon harian berdasarkan aktivitas pengguna,
khususnya konsumsi energi rumah tangga, makanan, dan perjalanan. Dengan fitur
pelacakan waktu nyata dan grafik tren mingguan, EcoTrack membuktikan

ECOSAUR

mencakup konsumsi energi listrik, makanan, limbah, dan transportasi, dapat
memberikan dampak perilaku yang signifikan (K.Arunasakthi, V.Akilandeswari,

Dengan melihat best

     penggunaan dengan antarmuka ramah pengguna.

Dengan demikian,

II.3.       Design Thinking

Setelah merancang
strategi pengembangan aplikasi mobile berbasis Android untuk menghitung jejak
karbon individu, tahapan selanjutnya adalah menerapkan pendekatan Design
Thinking guna memastikan solusi yang dikembangkan benar-benar relevan,
fungsional, dan sesuai dengan kebutuhan nyata pengguna. Pendekatan ini menjadi
langkah awal yang krusial sebelum masuk ke perancangan antarmuka dan pengalaman
pengguna (UI/UX), karena seluruh aspek visual dan interaksi dalam aplikasi
harus berakar pada pemahaman mendalam terhadap perilaku pengguna.

Design Thinking
merupakan metode pemecahan masalah yang menempatkan manusia sebagai pusat dari
setiap prosesnya. Pendekatan ini menggabungkan empati, kreativitas, dan
pemikiran kritis untuk menghasilkan solusi yang inovatif dan fungsional. Dengan
menekankan pentingnya pemahaman mendalam terhadap kebutuhan dan pengalaman
pengguna, Design Thinking telah banyak diterapkan dalam pengembangan produk dan
layanan digital guna menjawab permasalahan kompleks secara efektif (Telkom University, 2024). Prosesnya bersifat iteratif, kolaboratif lintas disiplin, dan
sangat fokus pada penciptaan nilai yang berdampak langsung bagi pengguna.

Sementara itu,
menurut (Harvard Business School Online, 2023) Design Thinking adalah
pendekatan berpikir dan bekerja yang mengutamakan inovasi berbasis pengguna.
Metode ini tidak hanya digunakan oleh desainer, tetapi juga oleh berbagai
profesional lintas industri untuk mengembangkan solusi yang kreatif dan relevan.
Berbeda dari pendekatan pemecahan masalah tradisional yang terfokus pada
analisis masalah, Design Thinking lebih menekankan pada pencarian solusi dengan
cara memahami siapa pengguna akhir dan bagaimana solusi tersebut akan berdampak
pada mereka. Hal ini dilakukan melalui proses iteratif yang mencakup
klarifikasi masalah, ideasi, pengembangan, hingga implementasi solusi.

Menurut panduan dari (Telkom University, 2024), tahapan Design Thinking secara umum meliputi lima langkah:

 Empathize :  memahami
     kebutuhan dan masalah pengguna melalui observasi dan wawancara;
 Define : merumuskan
     masalah yang tepat berdasarkan data yang dikumpulkan;
 Ideate :
     menghasilkan berbagai ide kreatif untuk solusi potensial;
 Prototype : membuat
     versi awal dari solusi untuk diuji;
 Test: menguji
     prototipe kepada pengguna dan mendapatkan umpan balik untuk penyempurnaan.

Sementara itu, (Harvard Business School Online, 2023) menekankan bahwa proses inovasi dalam Design
Thinking bergerak dari konkret ke abstrak dan kembali ke konkret, melalui empat
fase utama: clarify, ideate, develop, dan implement. Setiap fase bersifat
fleksibel dan memungkinkan pengulangan (iterasi) agar solusi yang dihasilkan
tidak hanya orisinal, tetapi juga praktis dan berdampak. Proses ini memastikan
bahwa solusi yang dikembangkan benar-benar memenuhi kebutuhan pengguna dengan
landasan empati dan validasi langsung di lapangan.

Dalam konteks
pengembangan aplikasi mobile pada penelitian ini, Design Thinking menjadi
pendekatan strategis yang diterapkan sejak tahap identifikasi kebutuhan
pengguna hingga evaluasi solusi. Dengan menerapkan prinsip-prinsip
human-centered design, aplikasi yang dikembangkan diharapkan mampu memberikan
pengalaman yang relevan dan mendorong perilaku berkelanjutan dalam mengelola
jejak karbon pribadi.

II.4.       Perancangan Antarmuka dan Pengalaman Pengguna

Setelah menerapkan
pendekatan Design Thinking untuk memahami kebutuhan dan perilaku
pengguna dalam konteks penggunaan energi rumah tangga, tahapan selanjutnya
adalah merancang antarmuka pengguna (User Interface/UI) dan pengalaman pengguna
(User Experience/UX) yang intuitif, fungsional, dan mampu mendorong
keterlibatan aktif dalam pelacakan jejak karbon.

II.4.1.  User Interface (UI)

User Interface (UI), atau antarmuka pengguna, merupakan titik interaksi antara
manusia dan mesin dalam konteks perangkat lunak maupun perangkat keras. UI
mencakup seluruh elemen visual dan interaktif yang memungkinkan pengguna
berkomunikasi dengan sistem, seperti tombol, ikon, menu, formulir, serta tata
letak halaman. UI berfungsi sebagai jembatan antara pengguna dan sistem agar
dapat menjalankan perintah serta memperoleh umpan balik secara efektif dan
efisien (Telkom University, 2024).

Menurut (Tech Target, 2024) , User interface didefinisikan sebagai the means by which
people (users) interact with a system, computer program or other complex tool.
Sementara itu, (Telkom University, 2024) menyatakan bahwa UI memungkinkan
pengguna untuk berinteraksi dengan sistem secara efisien dan efektif. Tanpa UI
yang baik, aplikasi maupun website akan mengalami kesulitan dalam
bersaing di era digital yang sangat kompetitif saat ini.

(Gould
& Lewis, 1985) dalam jurnalnya menyatakan bahwa desain antarmuka yang baik
harus memenuhi tiga prinsip utama, yaitu: (1) berfokus pada pengguna dan tugas
yang dijalankan, (2) melibatkan pengukuran empiris dalam penggunaan sistem, dan
(3) dilakukan secara iteratif agar dapat disesuaikan dengan kebutuhan pengguna
yang terus berkembang.

II.4.2.  User Experience (UX)

User Experience (UX)
adalah tentang keseluruhan pengalaman pengguna saat menggunakan aplikasi atau
website, yang mencakup interaksi, respons, kepuasan, hingga emosi yang
dirasakan oleh pengguna. Pengalaman ini dimulai dari saat pertama kali pengguna
membuka aplikasi hingga proses akhir dari interaksi tersebut. Dalam konteks
ini, UX tidak hanya mengukur seberapa mudah suatu sistem digunakan, tetapi juga
seberapa menyenangkan dan intuitif pengalaman tersebut bagi pengguna. UX
mencakup aspek-aspek seperti kegunaan (usability), aksesibilitas, dan pemenuhan
kebutuhan pengguna. Tujuan utama dari UX adalah menciptakan pengalaman yang
menyenangkan, mudah, dan intuitif, sehingga pengguna dapat berinteraksi dengan
sistem secara optimal dan tanpa hambatan (Telkom University, 2023).

Pendekatan terhadap
desain pengalaman pengguna menekankan pentingnya memahami perilaku, kebutuhan,
dan tujuan pengguna. Hal ini selaras dengan definisi dari ISO 9241-210:2019 (International Organization for Standardization, 2019) yang menyatakan bahwa UX merupakan
persepsi dan tanggapan pengguna yang dihasilkan dari penggunaan, atau
antisipasi penggunaan, terhadap suatu sistem atau produk. Aspek ini bersifat
subjektif dan sangat dipengaruhi oleh konteks penggunaan, karakteristik
pengguna, serta kualitas interaksi yang terjadi selama proses penggunaan.

Don Norman,
menjelaskan bahwa pengalaman pengguna terbentuk dari tiga lapisan utama, yaitu
visceral (respon awal atau kesan visual), behavioral (pengalaman selama
penggunaan), dan reflective (evaluasi dan pemaknaan setelah penggunaan). Ketiga
elemen ini membentuk persepsi pengguna secara keseluruhan terhadap suatu
sistem, produk, atau layanan (Norman, 2004). Dengan demikian, UX tidak dapat
dipisahkan dari faktor emosional dan psikologis yang menyertai pengguna selama
berinteraksi dengan sistem.

II.5.      
Business Process Model and Notation (BPMN)

Dalam merancang
aplikasi mobile yang bertujuan untuk membantu individu menghitung jejak karbon
dari aktivitas energi rumah tangga, penting untuk menggambarkan alur proses
secara menyeluruh. Salah satu pendekatan visual yang umum digunakan dalam
rekayasa perangkat lunak adalah Business Process Model and Notation (BPMN).
BPMN merupakan standar diagram yang digunakan untuk memodelkan proses bisnis
dalam bentuk visual yang dapat dipahami oleh berbagai pihak, mulai dari
pengguna, analis sistem, hingga pengembang (Object Management Group (OMG),
2013).

BPMN berfungsi
sebagai alat komunikasi visual antara perancang sistem dan pemangku
kepentingan. Dalam konteks aplikasi ini, BPMN digunakan untuk menggambarkan
urutan proses pengguna dalam menginput data konsumsi energi, validasi data,
perhitungan emisi karbon berdasarkan faktor emisi tertentu, hingga penyajian
hasil secara visual dan rekomendasi pengurangan emisi. Representasi ini penting
agar semua pihak yang terlibat memahami bagaimana sistem bekerja dari awal
hingga akhir, dan dapat memetakan potensi perbaikan dalam proses.

Elemen-elemen dalam
BPMN terdiri dari simbol grafis standar yang mewakili komponen utama dalam
sebuah proses bisnis. Elemen-elemen tersebut meliputi peristiwa (event),
aktivitas (activity), pengambilan keputusan (gateway), dan alur (flow). Setiap
simbol memiliki bentuk dan fungsi yang spesifik. Tabel berikut menyajikan
beberapa simbol BPMN beserta fungsinya (White, 2004):

Berdasarkan
elemen-elemen tersebut, alur proses bisnis dalam aplikasi ini dapat dijelaskan
secara naratif sebagai berikut: pengguna membuka aplikasi dan memilih jenis
aktivitas energi yang ingin dihitung, seperti konsumsi listrik, penggunaan gas
LPG, atau transportasi. Setelah itu, pengguna mengisi data terkait jumlah
energi yang dikonsumsi. Data tersebut kemudian divalidasi oleh sistem untuk
memastikan kelengkapan dan keakuratannya. Jika data valid, sistem akan
melanjutkan ke proses perhitungan emisi karbon dengan mengacu pada rumus dan
faktor emisi yang disesuaikan, seperti yang direkomendasikan oleh IPCC. Hasil
kalkulasi kemudian ditampilkan kepada pengguna dalam bentuk angka total emisi
CO&#8322;e serta grafik visual yang memudahkan pemahaman. Pengguna juga akan
mendapatkan saran tindakan untuk mengurangi emisi berdasarkan aktivitas mereka.

Keseluruhan proses
ini digambarkan secara visual dalam diagram BPMN agar pengembang dan perancang
dapat memahami jalannya sistem secara menyeluruh dan melakukan validasi
terhadap fungsionalitas yang dibutuhkan. BPMN juga berperan sebagai dokumentasi
teknis awal sebelum proses implementasi dilakukan. Penggunaan BPMN menjamin
bahwa sistem yang dibangun memiliki alur kerja yang terstruktur, terdefinisi
dengan baik, serta dapat diadaptasi dengan mudah pada tahap pengembangan dan
pengujian selanjutnya (Dumas, Rosa, Mendling, & Reijers, 2013) (White, 2004).

II.6.       Arsitektur Sistem

Perancangan antarmuka
dan pengalaman pengguna yang optimal dalam aplikasi mobile perlu didukung oleh
fondasi teknis yang kokoh, yaitu arsitektur sistem. Arsitektur sistem merupakan
struktur konseptual dari seluruh komponen perangkat lunak dan hubungan di
antara komponen tersebut yang memungkinkan sistem bekerja secara terpadu.
Menurut (Pressman & Maxim, Software Engineering: A Practitioner's Approach
9th Edition, 2019), arsitektur sistem dapat didefinisikan sebagai susunan
komponen perangkat lunak yang menunjukkan bagaimana sistem dipilah,
berkomunikasi, dan berevolusi dalam siklus hidupnya. Representasi arsitektur
ini tidak hanya menjadi kerangka pengembangan sistem, tetapi juga menjadi acuan
evaluasi keberlanjutan, keamanan, dan skalabilitas aplikasi.

Definisi yang lebih
formal menyatakan bahwa arsitektur perangkat lunak adalah struktur organisasi
dasar dari suatu sistem yang terdiri dari komponen-komponen perangkat lunak,
hubungan antar komponen tersebut, serta prinsip dan pedoman yang mengatur perancangan
dan evolusinya di masa mendatang (Voutyras, H. Bokhari, Tsuge, &
Palaiokrassas, 2021). Dalam konteks pengembangan aplikasi mobile untuk
menghitung jejak karbon individu dari aktivitas energi rumah tangga, arsitektur
sistem dirancang untuk menangani proses kalkulasi berbasis input pengguna,
pengelolaan data riwayat emisi, serta pengolahan hasil secara visual yang
responsif dan real-time.

Mengingat
kompleksitas sistem digital modern, sebuah arsitektur sistem umumnya tidak
direpresentasikan hanya dalam satu diagram tunggal, melainkan melalui berbagai view
atau sudut pandang. Masing-masing view menyajikan representasi sistem dari
perspektif tertentu, seperti fungsionalitas pengguna, struktur data, atau
proses bisnis yang terjadi. Pendekatan ini memungkinkan para pemangku
kepentingan baik pengembang, analis sistem, hingga pengguna akhir untuk memahami
bagian sistem yang relevan sesuai dengan perannya masing-masing (Voutyras,
H. Bokhari, Tsuge, & Palaiokrassas, 2021).

Arsitektur sistem
biasanya diorganisasi dalam bentuk pendekatan berlapis (layered architecture),
yang memisahkan sistem ke dalam beberapa lapisan utama sesuai fungsi dan
tanggung jawabnya. Pendekatan ini bertujuan untuk mendukung prinsip separation
of concerns, yakni memisahkan peran-peran teknis dalam sistem agar dapat
dikembangkan, diuji, dan dipelihara secara lebih efektif.

Lapisan pertama
adalah lapisan presentasi, yang berperan sebagai antarmuka utama antara
pengguna dan sistem. Di lapisan ini, seluruh interaksi pengguna seperti input
data, navigasi, dan tampilan hasil ditangani secara visual dan interaktif.
Lapisan kedua adalah lapisan logika aplikasi, yang bertugas memproses input,
menjalankan kalkulasi, serta mengatur alur logika sistem. Sementara itu,
lapisan data berperan menyimpan dan mengelola informasi penting yang dibutuhkan
oleh sistem, baik untuk kebutuhan kalkulasi maupun penyimpanan riwayat
aktivitas pengguna.

Dengan pendekatan
berlapis ini, setiap komponen sistem bekerja secara independen namun saling
terhubung secara fungsional. Hal ini tidak hanya meningkatkan fleksibilitas
pengembangan, tetapi juga memudahkan penerapan skenario pengujian, integrasi
komponen baru, serta pengembangan fitur di masa mendatang.

Untuk mendukung
dokumentasi arsitektur sistem secara visual dan standar, maka pada bagian
selanjutnya akan disajikan model pemodelan sistem menggunakan Unified Modeling
Language (UML), yang mencakup berbagai diagram penting seperti use case,
activity, dan class diagram, guna memvisualisasikan interaksi dan
struktur sistem secara menyeluruh.

II.7.       Unified Modeling Language (UML)

Unified Modeling
Language (UML) adalah bahasa pemodelan standar yang digunakan secara luas dalam
bidang rekayasa perangkat lunak untuk memvisualisasikan, merancang, dan
mendokumentasikan struktur serta perilaku sistem perangkat lunak. UML
dikembangkan oleh Grady Booch, Ivar Jacobson, dan James Rumbaugh pada awal
1990-an di bawah naungan Rational Software, dan kemudian diadopsi sebagai
standar oleh Object Management Group (OMG) pada tahun 1997 (Bennett
, McRobb, & Farmer, 2010).

UML menyediakan
seperangkat notasi grafis yang memungkinkan pengembang untuk merepresentasikan
berbagai aspek dari sistem perangkat lunak secara sistematis. Notasi ini tidak
hanya berguna dalam tahap desain awal, tetapi juga dapat dipakai sepanjang
siklus hidup perangkat lunak mulai dari spesifikasi kebutuhan hingga pengujian
dan pemeliharaan sistem. Dalam konteks pengembangan aplikasi penghitungan jejak
karbon, UML berperan penting untuk menjembatani pemahaman antara kebutuhan
pengguna, logika sistem, dan struktur teknis yang mendasari implementasi.

Menurut (Larman, 2005), UML memiliki dua kategori utama diagram, yaitu:

 Diagram struktural
      yang menggambarkan aspek statis dari sistem, seperti Class Diagram dan
     Component Diagram.
 Diagram perilaku 
     yang menggambarkan aspek dinamis dari sistem, seperti Use Case Diagram,
     Activity Diagram, dan Sequence Diagram.

Penggunaan diagram
UML dalam penelitian ini bertujuan untuk menggambarkan model-model konseptual
sistem yang dikembangkan agar proses implementasi dapat mengikuti struktur
desain yang terdefinisi dengan jelas. Beberapa diagram yang digunakan antara
lain:

 Use Case Diagram:
     Menampilkan fungsionalitas sistem dari perspektif pengguna (aktor).
     Diagram ini menjelaskan interaksi pengguna dengan sistem serta batasan
     sistem yang dikembangkan. Ini menjadi dasar penting dalam tahap
     perancangan kebutuhan fungsional sistem (Fowler, 2004).
 Activity Diagram:
     Menggambarkan alur kerja atau proses logika dari suatu fitur sistem.
     Diagram ini membantu memahami tahapan-tahapan operasional yang terjadi
     ketika pengguna menjalankan suatu aktivitas, seperti mengisi data energi
     harian atau melihat laporan emisi.
 Class Diagram:
     Mewakili struktur internal sistem berupa kelas-kelas objek, atribut, dan
     relasi antar kelas. Diagram ini menjadi kerangka utama dari arsitektur
     logika sistem berbasis objek.
 Sequence Diagram:
     Menunjukkan bagaimana objek dalam sistem saling berinteraksi dalam urutan
     waktu tertentu, biasanya untuk menggambarkan alur proses spesifik dari
     fitur sistem.

UML juga mendukung
prinsip multiple views architecture yang sejalan dengan standar
ISO/IEC/IEEE 42010:2011 tentang deskripsi arsitektur. Dengan menyediakan
beragam jenis diagram, UML memungkinkan perancang sistem untuk melihat sistem
dari berbagai sudut pandang baik dari sisi pengguna, pengembang, hingga analis
sistem sehingga meningkatkan pemahaman lintas peran terhadap sistem yang
dikembangkan (Object Management Group, 2017).

Dengan demikian, UML
bukan hanya alat bantu teknis, melainkan juga sarana komunikasi yang efektif
dalam proses kolaboratif pengembangan sistem. Dalam penelitian ini, pemanfaatan
UML menjadi bagian penting dari pendekatan perancangan sistem yang sistematis,
terdokumentasi, dan sesuai standar praktik terbaik dalam rekayasa perangkat
lunak.

II.8.       Software Development Life Cycle (SDLC)

Software Development
Life Cycle (SDLC) merupakan kerangka kerja sistematis yang digunakan untuk
mengelola proses pengembangan perangkat lunak dari tahap awal hingga tahap
pemeliharaan. SDLC bertujuan untuk menyediakan pendekatan terstruktur yang
dapat menjamin keberhasilan proyek perangkat lunak melalui perencanaan,
pengembangan, dan pemeliharaan yang terkendali (Sommerville, 2017). Pendekatan
ini telah lama diakui sebagai dasar dalam rekayasa perangkat lunak karena
membantu meningkatkan efisiensi, kualitas, dan keberlanjutan sistem yang
dikembangkan (S Pressman & R. Maxim, 2015) .

Menurut (S Pressman & R. Maxim, 2015), SDLC secara umum mencakup tujuh tahapan utama, yaitu:
perencanaan (planning), analisis kebutuhan (requirement analysis),
desain sistem (system design), implementasi (implementation),
pengujian (testing), penerapan (deployment), dan pemeliharaan (maintenance).
Setiap tahapan memiliki fungsi spesifik yang saling terkait dan membentuk
siklus iteratif yang memastikan perangkat lunak berkembang secara
berkelanjutan.

Dalam konteks
pengembangan aplikasi mobile untuk perhitungan jejak karbon rumah tangga,
penelitian ini mengadopsi model SDLC yang bersifat iteratif. Pendekatan
ini dinilai cocok untuk sistem yang melibatkan interaksi pengguna yang dinamis
serta kebutuhan akan perubahan dan penyesuaian berkelanjutan seiring proses
pengembangan (Fairley, 2009) .

Penjelasan tiap
tahapan SDLC sebagai berikut:

 Perencanaan
     (Planning)

Tahap
awal ini melibatkan identifikasi ruang lingkup sistem, penetapan tujuan proyek,
serta perkiraan sumber daya dan risiko. Aktivitas ini penting untuk menentukan
arah proyek secara keseluruhan dan menjadi dasar dari tahap-tahap selanjutnya (Sommerville, 2017).

 Analisis Kebutuhan
     (Requirement Analysis)

Proses
ini bertujuan untuk mengumpulkan kebutuhan fungsional dan non-fungsional sistem
dari pengguna, stakeholder, maupun studi literatur. Pengumpulan data dapat
dilakukan melalui wawancara, observasi, dan kuesioner (S Pressman & R.
Maxim, 2015).

 Desain Sistem
     (System Design)

Desain
sistem dilakukan untuk menjabarkan struktur teknis dan fungsionalitas sistem,
termasuk arsitektur perangkat lunak, struktur basis data, serta antarmuka
pengguna. Desain ini harus mampu memenuhi kebutuhan pengguna yang telah
diidentifikasi sebelumnya (Sommerville, 2017).

 Implementasi
     (Implementation)

Pada
tahap ini, pengkodean sistem dilakukan sesuai spesifikasi desain yang telah
disusun. Aplikasi dibangun menggunakan teknologi pilihan seperti Flutter untuk
tampilan dan Supabase sebagai backend. Implementasi dilakukan secara modular
untuk memudahkan pengujian dan pengembangan berkelanjutan (Fairley,
2009).

 Pengujian (Testing)

Tahap
pengujian bertujuan untuk mengevaluasi apakah sistem telah berjalan sesuai
harapan. Uji coba dilakukan baik secara unit maupun integrasi untuk
mengidentifikasi bug serta mengevaluasi kualitas dan stabilitas aplikasi (Sommerville, 2017).

 Penerapan
     (Deployment) 

Setelah
pengujian dinyatakan berhasil, sistem akan diimplementasikan di lingkungan
pengguna sesungguhnya. Proses ini dapat melibatkan peluncuran awal (soft
launch) untuk uji coba pengguna terbatas atau langsung diluncurkan secara
publik (S Pressman & R. Maxim, 2015).

 Pemeliharaan
     (Maintenance) 

Sistem
yang telah digunakan memerlukan proses pemantauan dan perbaikan berkala untuk
mengatasi bug, menambahkan fitur baru, dan meningkatkan performa berdasarkan
umpan balik pengguna (Fairley, 2009). Tahap ini merupakan bagian penting dari
siklus hidup perangkat lunak karena berkontribusi langsung terhadap
keberlanjutan sistem.

Pemilihan SDLC
sebagai pendekatan pengembangan memastikan bahwa proses pembangunan aplikasi
dilakukan secara sistematis, terdokumentasi, dan mudah dimonitor. Selain itu,
pendekatan ini juga memungkinkan penyesuaian secara iteratif, yang sangat
penting ketika merancang sistem berbasis pengguna (user-centered)
seperti aplikasi penghitung jejak karbon.

II.9.      
Teknologi yang digunakan

Dalam pengembangan
aplikasi mobile untuk menghitung jejak karbon dari aktivitas penggunaan energi
rumah tangga, pemilihan teknologi yang tepat merupakan faktor penting untuk
mendukung efisiensi, fleksibilitas, dan skalabilitas sistem. Penelitian ini
mengadopsi kombinasi antara Flutter sebagai framework pengembangan
antarmuka pengguna dan Supabase sebagai layanan backend. Kombinasi ini
memungkinkan proses pengembangan yang cepat, terstandarisasi, serta mendukung
arsitektur modern yang berbasis cloud dan real-time.

II.9.1.  Flutter (Dart)

Flutter merupakan
sebuah Software Development Kit (SDK) yang dikembangkan oleh Google dan
dirancang untuk membangun aplikasi mobile dengan kinerja tinggi. Framework ini
memungkinkan pengembangan aplikasi yang dapat dijalankan pada sistem operasi
Android dan iOS hanya dengan menggunakan satu basis kode (codebase) yang sama,
sehingga efisiensi waktu dan tenaga pengembangan dapat meningkat secara
signifikan. Flutter menggunakan bahasa pemrograman Dart yang dikenal cukup
mudah dipelajari, khususnya bagi pengembang yang telah familiar dengan Java
atau JavaScript. Selain itu, Flutter menawarkan pendekatan berbasis pemrograman
reaktif-fungsional, mesin rendering 2D bawaan, kumpulan widget yang siap pakai,
serta berbagai tools yang mempermudah proses desain dan pengembangan antarmuka
aplikasi yang modern dan responsif. Seluruh fitur tersebut menjadikan Flutter
sebagai salah satu teknologi pengembangan aplikasi mobile yang efisien dan
fleksibel dalam memenuhi kebutuhan sistem lintas platform (Muslim,
Sari, & Rahmayuda, 2022).

Sejak diluncurkan
pada tahun 2018, Flutter telah berkembang tidak hanya untuk aplikasi mobile,
tetapi juga mendukung platform web dan desktop (Windows, macOS, Linux). Salah
satu keunggulannya adalah kemampuan untuk merender tampilan secara langsung
menggunakan pustaka grafis Skia, yang menghasilkan antarmuka konsisten di
berbagai platform. Fitur seperti hot reload memungkinkan pengembang
melihat hasil perubahan kode secara real-time tanpa kehilangan status, sangat
mendukung proses iteratif dalam pengembangan antarmuka yang user-centered (Amazon Web Services, 2024). Dengan kemampuannya tersebut, Flutter sangat sesuai digunakan
dalam penelitian ini yang mengusung pendekatan design thinking untuk merancang
aplikasi kalkulator jejak karbon individu, karena memungkinkan iterasi
prototipe dan pengujian antarmuka secara cepat dan responsif.

II.9.2.  Supabase

Supabase adalah platform backend open-source yang dirancang
sebagai alternatif Firebase, menyediakan layanan autentikasi,
basis data relasional PostgreSQL, penyimpanan file, serta API real-time.
Platform ini semakin populer dalam pengembangan aplikasi modern karena
kemudahan integrasinya dan dukungan penuh terhadap standar web dan mobile
(Supabase, 2023).

Dalam penelitian ini,
Supabase digunakan sebagai solusi backend-as-a-service (BaaS) untuk
menyimpan data pengguna, mencatat hasil kalkulasi jejak karbon, serta menangani
autentikasi dan otorisasi pengguna. Layanan ini memungkinkan pengelolaan
data relasional secara fleksibel, sekaligus menyediakan fitur seperti JWT-based
authentication dan storage API yang dapat diakses langsung dari
aplikasi mobile berbasis Flutter.

Menurut (Arellano Serna, Torres, Torres, & López, 2024), Supabase mampu menyediakan
integrasi cepat antara antarmuka dan backend berkat API otomatis yang
dihasilkan dari struktur database PostgreSQL. Fitur ini memungkinkan
pengembang fokus pada logika aplikasi tanpa perlu membangun infrastruktur backend
secara manual.

Penelitian oleh (Jovanovi´c, 2025) juga menunjukkan bahwa Supabase dapat digunakan secara efektif
dalam pengembangan aplikasi mobile berskala menengah hingga besar. Dalam
studi tersebut, Supabase dimanfaatkan untuk membangun sistem e-commerce
berbasis React dan berhasil mengelola fitur-fitur real-time
seperti pencarian data dan komunikasi antar pengguna dengan performa yang
stabil. Hal ini menunjukkan kapabilitas Supabase sebagai backend
yang tangguh dan dapat diandalkan dalam lingkungan produksi.

Dengan kombinasi
antara kemudahan integrasi, dukungan dokumentasi yang baik, serta skalabilitas
berbasis cloud, Supabase merupakan pilihan backend yang
efisien untuk aplikasi modern, termasuk sistem yang memerlukan pencatatan
riwayat penggunaan energi secara individual seperti aplikasi penghitungan jejak
karbon.

II.9.3.  Pengujian dan Evaluasi

Setelah proses
pengembangan aplikasi selesai dilakukan berdasarkan pendekatan arsitektur
sistem dan siklus hidup pengembangan perangkat lunak, langkah selanjutnya
adalah melakukan pengujian dan evaluasi terhadap sistem yang telah dibangun.
Tahapan ini bertujuan untuk memastikan bahwa aplikasi berjalan sesuai dengan
spesifikasi yang ditentukan dan memberikan pengalaman pengguna yang optimal.
Untuk mencapai hal tersebut, penelitian ini menerapkan pendekatan pengujian
fungsional menggunakan metode Black Box Testing, serta mengevaluasi aspek
kegunaan (usability) aplikasi dengan menggunakan metode System Usability
Scale (SUS). Kedua pendekatan ini dipilih karena bersifat praktis,
terstandarisasi, dan telah terbukti efektif dalam menilai kualitas sistem dari
sisi teknis maupun persepsi pengguna.

II.9.3.1.    Blackbox Testing

Black Box Testing
merupakan metode pengujian perangkat lunak yang berfokus pada pengujian
fungsionalitas sistem tanpa memperhatikan struktur internal dari perangkat
lunak tersebut. Pendekatan ini digunakan untuk mengidentifikasi kesalahan atau
ketidaksesuaian yang mungkin terjadi dalam respons sistem terhadap masukan
tertentu, baik berupa kesalahan pada fungsi sistem, fitur yang tidak aktif,
maupun keluaran yang tidak sesuai dengan spesifikasi. Pengujian dilakukan
dengan memberikan berbagai data masukan, baik yang valid maupun tidak valid,
untuk melihat apakah sistem dapat mengenali, menerima, atau menolak input
tersebut secara tepat (S. Pressman & R. Maxim, 2019).

Salah satu prinsip
dari Black Box Testing adalah bahwa sistem dianggap sebagai kotak hitam yang
hanya dapat diakses dari luar; penguji tidak perlu mengetahui bagaimana proses
logika dalam kode berjalan. Yang diperhatikan hanyalah bagaimana sistem merespon
terhadap input yang diberikan dan apakah output yang dihasilkan sesuai dengan
harapan (J. Myers, Sandler, & Badgett, 2015).

Dalam pelaksanaannya,
terdapat beberapa teknik umum dalam Black Box Testing yang sering digunakan,
antara lain:

 Equivalence
     Partitioning

Teknik
ini membagi data masukan menjadi beberapa kelompok atau partisi yang dianggap
mewakili kondisi input yang serupa. Dari masing-masing partisi, diambil satu
atau dua nilai sebagai uji representatif. Tujuannya adalah untuk mengurangi
jumlah kasus uji sambil tetap mencakup berbagai kondisi input penting. Dalam
konteks aplikasi penghitungan jejak karbon, misalnya, pengguna dapat menginput
nilai konsumsi listrik dalam satuan kWh. Teknik ini membantu menguji apakah
sistem dapat menangani input dalam kisaran wajar (misalnya 1001000 kWh) serta
menolak nilai di luar batas (misalnya input teks atau angka negatif).

 Boundary Value
     Analysis

Metode
ini menguji batas nilai minimum dan maksimum dari input untuk mendeteksi
kesalahan yang biasanya muncul di tepi rentang nilai. Misalnya, jika aplikasi
hanya menerima nilai konsumsi LPG antara 150 liter, maka pengujian dilakukan
pada nilai 0, 1, 50, dan 51 untuk melihat respons sistem.

 State Transition
     Testing

Digunakan
untuk menguji transisi status sistem berdasarkan input atau tindakan pengguna.
Ini cocok diterapkan saat menguji navigasi antarmuka aplikasi dan alur
pengisian data hingga pelaporan hasil jejak karbon.

 Cause-Effect
     Graphing

Merupakan
teknik yang menggambarkan hubungan logis antara input dan output dalam bentuk
graf. Cocok untuk sistem yang memiliki banyak aturan kondisi seperti jika-maka.

 All-Pairs Testing
     dan Orthogonal Array Testing

Digunakan
untuk menguji kombinasi input dari berbagai parameter, agar semua pasangan
kombinasi diuji minimal sekali. Ini berguna bila terdapat banyak parameter
input dalam satu fitur.

 Fuzzing

     Teknik ini menginjeksi data rusak atau tidak lazim untuk melihat apakah
     sistem dapat bertahan dari input yang ekstrem atau cacat. Walau umumnya
     digunakan untuk uji keamanan, teknik ini dapat memberikan wawasan
     ketahanan sistem terhadap input tidak normal.

Menurut (J.
Myers, Sandler, & Badgett, 2015), teknik pengujian ini efektif karena mampu
mengungkap berbagai kesalahan potensial dengan pendekatan sistematis, dan
sangat cocok digunakan dalam pengembangan sistem informasi yang membutuhkan
keandalan tinggi.

II.9.3.2.       Usability dengan System Usability Scale (SUS)

Usability atau
kegunaan sistem merupakan aspek penting dalam pengembangan perangkat lunak,
terutama dalam desain antarmuka pengguna aplikasi mobile. Menurut ISO 9241-11,
usability didefinisikan sebagai sejauh mana suatu produk dapat digunakan oleh
pengguna untuk mencapai tujuan tertentu secara efektif, efisien, dan memuaskan
dalam suatu konteks penggunaan tertentu.  (Nielsen, 1993) mengidentifikasi lima
komponen utama dalam usability, yaitu learnability (kemudahan pengguna dalam
mempelajari sistem), efficiency (kecepatan pengguna dalam menyelesaikan tugas),
memorability (kemampuan pengguna mengingat kembali penggunaan sistem setelah
jeda waktu), error (jumlah dan tingkat kesalahan yang dibuat pengguna serta
kemudahan mengatasinya), dan satisfaction (tingkat kenyamanan pengguna dalam
menggunakan sistem). Dalam konteks aplikasi mobile, usability menjadi semakin
penting karena perangkat ini digunakan secara cepat, fleksibel, dan dalam
berbagai kondisi yang dinamis.

Salah satu metode
evaluasi usability yang paling banyak digunakan dan telah teruji secara ilmiah
adalah System Usability Scale (SUS). SUS dikembangkan oleh John Brooke pada
tahun 1986 sebagai alat ukur yang sederhana, cepat, dan andal untuk menilai
persepsi pengguna terhadap kegunaan suatu sistem. SUS terdiri dari sepuluh
pernyataan yang dijawab oleh pengguna dengan menggunakan skala Likert lima
poin, dari 1 (sangat tidak setuju) hingga 5 (sangat setuju) (Brooke,
1995). Keunggulan SUS terletak pada kemampuannya menggabungkan persepsi
kognitif dan afektif pengguna terhadap sistem menjadi satu skor numerik yang
dapat diinterpretasikan secara komparatif. SUS bersifat teknologi-agnostik,
sehingga dapat digunakan untuk menilai sistem desktop, perangkat mobile,
aplikasi web, bahkan produk perangkat keras. Berikut adalah daftar 10
pernyataan dalam System Usability Scale dalam versi bahasa Indonesia:

Berbagai penelitian
telah menguatkan efektivitas dan fleksibilitas penggunaan SUS dalam evaluasi
aplikasi mobile. (Fergo & Ratnasari, 2023) menggunakan SUS untuk
mengevaluasi aplikasi finansial Octo Mobile dan menemukan bahwa metode ini
efektif dalam mengidentifikasi bagian-bagian antarmuka yang membingungkan
pengguna. (Rohandi, Husain, & Bay, 2021) menggunakan SUS untuk menilai
aplikasi pembelajaran dan menemukan bahwa nilai SUS yang tinggi berkorelasi
dengan peningkatan kepuasan pengguna. (Islam, Khan, Islam, & Rownok, 2021)
dalam pengembangan aplikasi kesehatan mental juga menerapkan SUS untuk
menentukan kelayakan sistem dari perspektif pengguna, sedangkan (Hyzy,
et al., 2022) melalui studi meta-analisis menemukan bahwa sebagian besar
aplikasi kesehatan digital memiliki skor SUS antara 75 hingga 80, yang
menandakan penerimaan pengguna yang baik. Kortum dan Sorber (2015) juga
menyatakan bahwa nilai usability suatu aplikasi dapat bervariasi antara versi
tablet dan ponsel meskipun desain antarmuka serupa, dan SUS menjadi alat yang
efektif untuk membandingkan keduanya.

Dalam penelitian ini,
perancangan aplikasi mobile untuk menghitung dan menyajikan jejak karbon
individu berdasarkan aktivitas konsumsi energi sehari-hari membutuhkan
pendekatan usability yang cermat. Informasi mengenai emisi karbon cenderung
bersifat teknis dan kompleks, sehingga penting untuk disampaikan melalui
antarmuka yang sederhana, mudah dipahami, dan menarik. Evaluasi usability
menggunakan metode System Usability Scale akan memberikan gambaran sejauh mana
pengguna merasa nyaman, efisien, dan percaya diri dalam menggunakan aplikasi
ini. Hasil evaluasi ini tidak hanya akan mendukung validitas fungsionalitas
sistem, tetapi juga menjadi landasan perbaikan desain pada tahap iteratif
berikutnya. Oleh karena itu, penggunaan SUS dalam penelitian ini akan menjadi
komponen penting dalam memastikan bahwa aplikasi yang dikembangkan benar-benar
mampu digunakan dan diterima oleh masyarakat luas secara optimal.

BAB III METODE PENYELESAIAN MASALAH

III.1     
Sistematika Penyelesaian Masalah

Sistematika
penyelesaian masalah dalam penelitian ini mengadopsi pendekatan iteratif yang
terinspirasi dari prinsip-prinsip Agile, dengan penyesuaian untuk konteks
penelitian individual. Pemilihan pendekatan iteratif didasarkan pada
karakteristik penelitian yang berfokus pada refactoring dan migrasi sistem, di
mana kebutuhan perubahan dan penyesuaian dapat muncul secara dinamis selama
proses berlangsung. Pendekatan ini memungkinkan pengembangan dilakukan dalam
iterasi-iterasi pendek dengan evaluasi berkelanjutan, sehingga setiap tahap
perbaikan dapat divalidasi sebelum melanjutkan ke tahap berikutnya.

Proses iteratif dalam
penelitian ini terdiri dari lima tahapan utama: Persiapan dan Analisis Awal,
Perencanaan Iterasi, Pelaksanaan Refactoring dan Migrasi, Evaluasi dan
Pengujian, serta Refleksi dan Perbaikan. Setiap iterasi dirancang dengan durasi
1-2 minggu, dengan fokus pada penyelesaian item-item perbaikan yang telah
diprioritaskan berdasarkan tingkat urgensi dan dependensi teknis.

Tahap 1: Persiapan
dan Analisis Awal

Tahap pertama adalah
penyusunan daftar pekerjaan (backlog), yaitu inventarisasi lengkap semua
pekerjaan yang perlu dilakukan dalam penelitian ini. Proses dimulai dengan
evaluasi mendalam terhadap aplikasi Carbonize versi sebelumnya untuk
mengidentifikasi masalah-masalah teknis yang ada. Evaluasi dilakukan melalui
code review untuk mendeteksi code smells seperti duplikasi kode, low cohesion,
high coupling, dan kurangnya modularisasi. Selain itu, dilakukan analisis
terhadap struktur data Firebase yang digunakan untuk memahami skema data,
relasi antar entitas, dan pola akses data.

Hasil evaluasi ini
kemudian diterjemahkan menjadi backlog items yang mencakup:

 Identifikasi dan
     dokumentasi code smells yang perlu diperbaiki 
 Pemetaan struktur
     data Firebase yang akan dimigrasikan 
 Perancangan skema
     basis data relasional 
 Perancangan
     arsitektur backend RESTful API 
 Strategi migrasi
     data dari Firebase ke DBMS 
 Perancangan
     struktur modular kode frontend 
 Implementasi
     refactoring per modul 
 Pengujian dan
     validasi sistem

Setiap backlog item
diberi prioritas berdasarkan tingkat urgensi dan dependensi antar-item.
Prioritas tertinggi diberikan pada pembangunan backend API karena merupakan
fondasi infrastruktur yang harus tersedia terlebih dahulu sebelum frontend
dapat di-refactor untuk menggunakan arsitektur baru. Hal ini sejalan dengan
prinsip pengembangan sistem terdistribusi di mana backend sebagai sumber data
harus stabil sebelum dilakukan integrasi dengan client.

Tahap 2: Perencanaan Iterasi

Tahap kedua adalah
perencanaan detail untuk setiap iterasi yang akan dilakukan. Dalam setiap sesi
perencanaan, backlog items dengan prioritas tertinggi dipilih untuk dikerjakan
dalam iterasi tersebut. Pembagian iterasi dalam penelitian ini dirancang dengan
mempertimbangkan dependensi teknis dan alur kerja yang logis, yaitu:

Iterasi 1: Perancangan
dan Implementasi Backend API 

Iterasi pertama
difokuskan pada pembangunan backend sebagai fondasi sistem baru. Rasional
pemilihan backend sebagai prioritas pertama adalah:

1.     Dependensi Teknis: Frontend memerlukan backend API yang sudah berfungsi
untuk dapat diuji dan diintegrasikan. Membangun backend terlebih dahulu
memungkinkan pengembangan frontend dilakukan dengan target integrasi yang
jelas.

2.    
Stabilitas Infrastruktur: Migrasi basis data dari
Firebase ke DBMS relasional merupakan perubahan fundamental yang mempengaruhi
seluruh arsitektur sistem. Memastikan backend stabil terlebih dahulu
meminimalkan risiko perubahan besar di tengah pengerjaan frontend.

3.     Validasi Skema Data: Implementasi backend memungkinkan validasi awal
terhadap desain skema database dan API endpoints sebelum frontend dikembangkan,
sehingga menghindari rework yang signifikan.

Fokus pada iterasi
ini meliputi:

·      
Perancangan skema basis data relasional dengan
mempertimbangkan normalisasi data dan integritas referensial

·      
Implementasi backend menggunakan Laravel dengan
arsitektur RESTful API yang terstruktur

·      
Pembuatan Data Access Layer melalui Laravel
Eloquent ORM

·      
Implementasi RESTful API endpoints untuk setiap
operasi CRUD

·      
Migrasi data dari Firebase ke MySQL/PostgreSQL

·      
Pengujian API menggunakan tools seperti Postman
untuk memastikan seluruh endpoints berfungsi dengan benar

Iterasi 2: Refactoring
Frontend dan Integrasi dengan Backend API 

Iterasi kedua
dilakukan setelah backend API selesai dan stabil. Rasional urutan ini adalah
frontend dapat di-refactor sekaligus diintegrasikan dengan backend baru dalam
satu iterasi yang kohesif, menghindari duplikasi effort yang terjadi jika
refactoring dilakukan terpisah dari integrasi.

Fokus pada iterasi
ini meliputi:

·      
Refactoring struktur kode frontend dengan
menerapkan organisasi berbasis fitur (feature-based structure)

·      
Restructuring kode menjadi modul-modul yang
terpisah dan independen, mirip dengan konsep modules, di mana setiap fitur
utama aplikasi dikelompokkan dalam satu folder yang berisi semua komponen,
services, dan state management yang terkait dengan fitur tersebut

·      
Pemisahan concerns dengan prinsip pragmatis:
memisahkan UI components, business logic (services/repositories), dan state
management secukupnya untuk memudahkan pemeliharaan dan pengembangan
selanjutnya, tanpa over-engineering dengan layer yang terlalu kompleks

·      
Implementasi koneksi antara frontend Flutter dengan
backend Laravel REST API

·      
Penggantian seluruh integrasi Firebase dengan
pemanggilan HTTP ke backend API

·      
Penyesuaian model data frontend agar sesuai dengan
response API backend

·      
Penghapusan dependensi Firebase dari aplikasi
Flutter

·      
Penerapan struktur folder yang jelas dan konsisten
untuk setiap modul/fitur, sehingga pengembang baru dapat dengan mudah menemukan
dan memahami kode yang terkait dengan fitur tertentu.

Refactoring dilakukan
dengan menerapkan prinsip-prinsip Clean Code yang esensial seperti penamaan
yang jelas, fungsi yang tidak terlalu panjang, dan menghindari duplikasi kode.
Pendekatan yang diambil adalah pragmatic refactoring: fokus pada peningkatan maintainability
yang signifikan tanpa membuat struktur yang terlalu kompleks atau
over-engineered.

Iterasi 3: Pengujian
dan Validasi

Iterasi ketiga fokus
pada pengujian komprehensif dan validasi sistem secara menyeluruh setelah
seluruh komponen selesai diintegrasikan. Pemisahan tahap pengujian ke iterasi
terpisah memberikan waktu yang cukup untuk melakukan pengujian mendalam tanpa
terburu-buru, memastikan kualitas deliverable akhir.

Fokus pada iterasi
ini meliputi:

Pengujian
Fungsionalitas dilakukan menggunakan metode Black Box Testing dan regression
testing untuk memastikan bahwa refactoring tidak mengubah perilaku eksternal
aplikasi (behavior preservation). Setiap fitur utama aplikasi diuji untuk
memastikan masih berfungsi sesuai dengan spesifikasi awal. Pengujian mencakup:

·      
Pengujian seluruh fitur aplikasi secara end-to-end

·      
Verifikasi integrasi frontend-backend pada berbagai
skenario penggunaan

·      
Pengujian edge cases dan error handling

·      
Validasi keamanan API (authentication dan
authorization)

Pengukuran Metrik
Kualitas Kode Sesuai dengan batasan penelitian yang
telah ditetapkan pada Bab I, evaluasi difokuskan pada aspek maintainability
dengan menggunakan metrik perangkat lunak yang terukur

1.     Cyclomatic Complexity: Mengukur kompleksitas logika kode berdasarkan
jumlah jalur independen dalam program. Semakin rendah nilai cyclomatic
complexity, semakin mudah kode untuk dipahami dan dipelihara.

2.    
Code Duplication: Mengidentifikasi persentase kode
yang terduplikasi dalam codebase. Refactoring bertujuan mengurangi duplikasi
melalui ekstraksi method dan penerapan prinsip DRY (Don't Repeat Yourself).

3.     Maintainability Index: Metrik komposit yang menggabungkan beberapa
faktor untuk menghasilkan skor kemudahan pemeliharaan. Nilai yang lebih tinggi
mengindikasikan kode yang lebih mudah dipelihara.

Pengukuran metrik
dilakukan menggunakan tools analisis statis seperti Dart Code Metrics untuk
kode Flutter. Hasil pengukuran dibandingkan antara kondisi sebelum dan sesudah
refactoring untuk mengevaluasi peningkatan kualitas. Dalam setiap iterasi,
ditetapkan kriteria penyelesaian (Definition of Done) yang mencakup: kode telah
di-review secara mandiri, pengujian dasar berhasil, dan dokumentasi telah
dibuat.

Tahap 3:  Pelaksanaan Refactoring dan
Migrasi

Tahap ketiga adalah
pelaksanaan pekerjaan yang telah direncanakan sesuai dengan pembagian iterasi.
Pada tahap ini, dilakukan implementasi sesuai dengan fokus masing-masing

praktik-praktik terbaik pengembangan perangkat lunak dengan dokumentasi yang

Tahap 4: Evaluasi dan
Pengujian

evaluasi hasil kerja yang telah diselesaikan dalam setiap iterasi. Evaluasi

yang telah ditetapkan. Detail pengujian dan evaluasi telah dijelaskan pada deskripsi
Iterasi 3 di Tahap 2.

Tahap 5: Refleksi dan
Perbaikan Berkelanjutan

Tahap kelima adalah
refleksi terhadap proses yang telah dilakukan dalam setiap iterasi untuk

seperti efektivitas strategi migrasi backend yang digunakan, kualitas
implementasi refactoring frontend, kendala yang dihadapi selama integrasi

Hasil refleksi
digunakan untuk menyusun perbaikan yang akan diterapkan pada iterasi

Evaluasi Akhir

·

·

Dengan menggunakan

IV.2.1 Analisis Kondisi Awal Aplikasi

Analisis kondisi awal

original. Aplikasi ini dikembangkan menggunakan framework Flutter untuk
frontend mobile dan Firebase sebagai Backend-as-a-Service (BaaS).
Pemahaman yang mendalam terhadap kondisi awal diperlukan untuk mengidentifikasi

A.    Struktur Folder Aplikasi Original

Struktur folder
proyek menggambarkan bagaimana kode diorganisasikan dalam aplikasi. Analisis
struktur folder penting untuk memahami pola arsitektur yang digunakan dan
mengidentifikasi potensi masalah organisasi kode, aplikasi Carbonize memiliki
struktur folder sebagai berikut:

Gambar IV- 1  Struktur Aplikasi Saat Ini

Berdasarkan gambar
diatas, ditemukan bahwa aplikasi saat ini memiliki total estimasi lebih dari
11.000 baris kode yang tersebar tidak merata pada 22 file Dart. Analisis lebih
lanjut menunjukkan tiga masalah fundamental.

1.    
Fat UI & Low Cohesion: Berkas pada folder screens/ menggabungkan logika bisnis dan tampilan
antarmuka secara erat (tightly coupled). Hal ini terbukti dari berkas
calculator_screen.dart (4.409 LOC) yang menangani terlalu banyak tanggung jawab

2.    
Fragile Maintenance: Ketiadaan pemisahan concern (Separation of Concerns)
menyebabkan setiap perubahan kecil pada logika kalkulasi berisiko merusak
tatanan UI, karena tidak adanya batasan modul yang jelas.

3.    
Absence of Modularization: Struktur folder saat ini hanya membagi berkas berdasarkan tipe teknis
(seperti screens/ atau services/), bukan berdasarkan fungsionalitas fitur.

Transformasi
Arsitektur guna mengatasi kendala tersebut, dalam tahap pengembangan ini
dilakukan transformasi dari struktur monolitik menuju Feature-Driven Modular
Architecture. Melalui arsitektur ini, aplikasi didekonstruksi menjadi
modul-modul fungsional mandiri (seperti Auth Module, Calculator
Module, dan Profile Module). Setiap modul dirancang memiliki
enkapsulasi terhadap manajemen logika bisnis dan komponen antarmukanya sendiri.
Implementasi ini bertujuan untuk :

·      
Meningkatkan Keterbacaan (Readability): Struktur
kode menjadi lebih terorganisir karena setiap fungsionalitas terlokalisasi
dalam satu direktori modul.

·      
Mempermudah Pemeliharaan (Maintainability):
Perubahan pada satu fitur tidak akan berdampak secara tidak terduga pada fitur
lainnya (low coupling).

·      
Optimalisasi Pengujian: Memungkinkan dilakukannya
pengujian unit (unit testing) secara terisolasi pada tiap-tiap komponen modul.

IV.2.4 Analisis Logika Kalkulasi Emisi

Salah satu aspek
kritis dalam proses migrasi adalah memastikan bahwa logika kalkulasi emisi
karbon menghasilkan nilai yang identik dengan sistem original. Kesalahan dalam
implementasi logika kalkulasi akan menyebabkan inkonsistensi data dan
mengurangi kepercayaan pengguna terhadap sistem baru. Oleh karena itu, analisis
mendalam terhadap setiap rumus kalkulasi yang digunakan dalam sistem Firebase
menjadi sangat penting. Berdasarkan analisis terhadap file
lib/services/emission_service.dart, teridentifikasi empat jenis kalkulasi yang
digunakan dalam aplikasi. Masing-masing kalkulasi memiliki karakteristik dan
rumus yang berbeda sesuai dengan kategori konsumsinya.

A.    Kalkulasi Food & Packaging (Nilai Tetap)

Kalkulasi untuk
kategori food and packaging dengan nilai tetap merupakan
kalkulasi yang paling sederhana. Metode ini menggunakan faktor emisi yang telah
ditentukan sebelumnya untuk setiap jenis item, kemudian dikalikan dengan
kuantitas yang dikonsumsi.

Gambar IV- 3 Kondisi Awal Kalkulasi kategori Food &
Packaging

Kode di atas menunjukkan
implementasi kalkulasi nilai tetap dalam Firebase.
Fungsi getFixedEmissionValues() mengambil data faktor emisi dari
Firestore, kemudian nilai faktor emisi untuk item yang dipilih dikalikan dengan
kuantitas input. Rumus matematis yang digunakan adalah:

Sebagai contoh, untuk
item &quot;Cardboard Boxes&quot; dengan faktor emisi 0.85 kg CO&#8322;e/kg dan
kuantitas 2.5 kg, maka emisi yang dihasilkan adalah 0.85 × 2.5 = 2.125 kg
CO&#8322;e.

B.    Kalkulasi Food & Packaging (Climatiq API)

Untuk item makanan
tertentu yang memerlukan data emisi yang lebih akurat, aplikasi menggunakan
integrasi dengan Climatiq API. Climatiq merupakan layanan yang menyediakan data
faktor emisi karbon untuk berbagai aktivitas berdasarkan activity ID yang
spesifik.

Gambar IV- 4 Kondisi Awal Kalkulasi Food & Packaging yang

Kode di atas
menunjukkan bahwa untuk item yang menggunakan Climatiq, aplikasi
menyimpan activityID (bukan nilai numerik) di Firestore.
Ketika kalkulasi dilakukan, aplikasi mengirim HTTP POST request ke
Climatiq API dengan activity ID dan kuantitas, kemudian API
mengembalikan nilai emisi yang dihitung berdasarkan metodologi standar
internasional.

merupakan praktik yang tidak aman karena key dapat diekstrak dari aplikasi yang
terinstall.

IV.3.4 Definition of Done

Untuk menjamin
kualitas output pada setiap iterasi dan menghindari ambiguitas dalam
penyelesaian tugas, ditetapkan standar Definition of Done (DoD) sebagai
berikut:

1.     Standar
Kode: Kode program telah selesai ditulis dengan mengikuti coding standards
PSR-12 (Laravel).

2.     Aksesibilitas:
Endpoint dapat diakses melalui protokol HTTP dengan method yang sesuai (GET,
POST, PUT, DELETE).

3.     Validasi
Data: Fungsi validasi input berjalan dengan benar, memberikan pesan kesalahan
(error message) yang informatif saat data tidak valid.

4.     Konsistensi
Response: Format response JSON seragam untuk seluruh endpoint (mengikuti
standar struktur yang ditetapkan).

5.     Integritas
Sistem: Tidak ditemukan error atau exception pada log server selama proses
pengujian.

6.     Akurasi
Kalkulasi: Hasil perhitungan emisi pada backend telah divalidasi dan sesuai
dengan rumus manual yang ditetapkan.

7.     Verifikasi
Eksternal: Seluruh endpoint telah lulus uji coba melalui perangkat lunak
Postman dengan status kode 200 OK (atau 201 Created).\

organisasi dan ketergantungan antar komponen perangkat lunak. Sistem ini
memisahkan tanggung jawab antara client-side (Flutter) dan server-side
(Laravel) untuk memastikan modularitas dan kemudahan pemeliharaan.

UI Layer (Screens): Mengelola interaksi langsung dengan
pengguna.

·       
Local Storage (SharedPreferences): Digunakan untuk

·       
Services Layer: Bertindak sebagai jembatan yang
menghubungkan lapisan UI dan penyimpanan lokal untuk melakukan permintaan data

2.     Laravel
Backend Application:

·       
Routes (api.php): Menentukan titik masuk (endpoint)
untuk setiap permintaan yang datang dari frontend.

·       
Middleware (Sanctum, CORS): Melakukan filtrasi keamanan,
termasuk pemeriksaan token autentikasi melalui Laravel Sanctum dan pengaturan
kebijakan akses lintas asal (cross-origin).

·       
Controllers: Mengatur alur logika aplikasi untuk entitas
Auth, User, dan Emission. Komponen ini menerima input, memvalidasinya, dan
meneruskannya ke lapisan layanan.

·       
Services Layer: Berisi logika bisnis inti yang terisolasi,
yang mencakup ClimatiqService untuk integrasi API eksternal dan Emission
Calculation Logic untuk perhitungan emisi karbon (makanan, bahan bakar, dan
transportasi).

·       
Models (Eloquent ORM): Merepresentasikan skema data dan
relasi antar entitas, yang mencakup model User, EmissionFactorCategory,
EmissionFactorItem, dan ConsumptionEntry.

3.     MySQL
Database:

·       
Database: Merupakan lapisan penyimpanan data akhir yang
dikelola oleh Models melalui driver basis data untuk menjamin
integritas dan keamanan data.

IV.4.5          
Implementasi Climatiq Service

Logika integrasi API Climatiq

kalkulasi berbasis berat (Food & Packaging) dan
calculateFuelEmissions()untuk kalkulasi berbasis volume bahan bakar.

Beberapa keputusan
desain pada service ini:

·      
API Key diambil melalui config() yang membaca environment
variable, sehingga tidak pernah hard-code dalam kode sumber.

·      
Logging diterapkan pada setiap request dan response untuk
keperluan debugging dan audit trail

·      
Error handling menggunakan try-catch dan mengembalikan
format response yang konsisten (['success' =&gt; true/false, ...]),
sehingga controller bisa menangani error secara seragam

·      
Parameter yang fleksibel dimana method calculateEmissions()
menerima parameter unit dan parameterType sehingga bisa digunakan untuk
berbagai jenis input (berat, volume, jarak).

Hanya endpoint
register dan login yang dapat diakses tanpa token. Seluruh route
lainnya dilindungi oleh dua middleware: auth:sanctum untuk validasi token
Sanctum dan custom.token untuk validasi tambahan. Request tanpa token
valid akan langsung ditolak dengan status HTTP 401 (Unauthenticated)
sebelum kode controller dieksekusi. Penggunaan middleware
auth:sanctum juga berperan dalam pencegahan kerentanan Broken Object Level
Authorization (BOLA). Dengan middleware ini, setiap request
yang masuk sudah teridentifikasi penggunanya melalui token, sehingga controller
dapat memfilter data berdasarkan user_id dari token untuk memastikan
pengguna A tidak dapat mengakses atau menghapus data konsumsi milik pengguna B
hanya dengan mengganti parameter id pada URL. Penggunaan prefix /v1/
pada seluruh route bertujuan untuk mendukung API versioning yang
dimana jika di kemudian hari diperlukan perubahan yang tidak backward-compatible,
versi baru (/v2/) bisa ditambahkan tanpa mengganggu klien yang masih
menggunakan versi lama.

 

 

 

IV.5       Tahap
4: Evaluasi dan Pengujian (Iterasi 1)

Setelah implementasi
selesai, tahap ini bertujuan untuk memvalidasi bahwa seluruh komponen yang

krusial untuk memastikan kualitas deliverable sebelum
dilanjutkan ke iterasi berikutnya.

IV.5.1 Metodologi Pengujian

Pengujian dilakukan
dengan metode Black Box Testing, yang berfokus pada fungsionalitas
tanpa memeriksa struktur internal kode. Pendekatan ini sesuai untuk pengujian
API karena yang divalidasi adalah kesesuaian antara request yang
dikirim dan response yang diterima, bukan bagaimana kode di
dalamnya bekerja.

Tiga pendekatan pengujian yang
diterapkan:

 Functional Testing,
     yaitu menguji setiap endpoint secara individual untuk
     memastikan berfungsi sesuai spesifikasi, termasuk validasi input dan
     format response.
 Integration Testing,
     yaitu menguji interaksi antar komponen seperti controller, service, model,
     dan database secara terintegrasi.
 Comparison Testing,
     yaitu membandingkan hasil kalkulasi Laravel dengan Firebase untuk
     memastikan konsistensi data setelah migrasi.

Pengujian dilakukan menggunakan
Postman v10.x untuk pengiriman request dan validasi response, serta MySQL
Workbench untuk verifikasi data di database.

Perlu dicatat bahwa pengujian ini
menggunakan pendekatan equivalence partitioning, di mana skenario dikelompokkan
berdasarkan jalur logika kalkulasi. Item-item dalam partisi yang sama (misalnya
seluruh jenis BBM pada kategori Fuel Consumption) menggunakan rumus identik
sehingga cukup diwakili oleh satu sampel representatif per partisi. Pendekatan
ini umum digunakan dalam Black Box Testing untuk mengurangi jumlah test case
tanpa mengurangi cakupan pengujian secara signifikan.

 Dokumentasi data

     membaca data langsung dari console Firebase.

     mencegah rework. Beberapa jam terbuang karena asumsi
     yang keliru tentang skema database. Verifikasi menyeluruh di
     awal bisa menghemat waktu dan menghindari migration tambahan
     yang sebenarnya tidak perlu.
 Pengujian dengan data riil
     memastikan akurasi. Validasi menggunakan data ekspor Firebase
     asli membuktikan bahwa migrasi berhasil dengan akurasi 100%. Pendekatan
     ini lebih meyakinkan dibandingkan pengujian dengan data buatan.
 Naming convention bisa
     menggantikan tabel relasi. Suffix &quot;(Emission)&quot;
     dan &quot;(Passengers)&quot; terbukti efektif untuk menghubungkan data
     berpasangan tanpa memerlukan infrastruktur tabel relasi tambahan. Meskipun
     solusi ini tepat untuk skala data saat ini (40 item), perlu dievaluasi
     kembali jika data bertambah signifikan.

IV.8      
Tahap 6: Perencanaan Iterasi 2

Setelah backend
Laravel API berhasil dibangun dan divalidasi pada Iterasi 1, fokus pengerjaan
beralih ke sisi frontend. Iterasi 2 bertujuan untuk melakukan refactoring
arsitektur frontend Flutter dan mengintegrasikannya dengan backend
API yang telah tersedia, sehingga ketergantungan langsung terhadap Firebase
dapat dihilangkan sepenuhnya.

IV.8.1 Tujuan
Iterasi 2

Berdasarkan backlog yang telah disusun pada
akhir Iterasi 1 (Tabel IV-25), Iterasi 2 memiliki tiga tujuan utama.

·      
Tujuan pertama adalah reorganisasi arsitektur frontend. Struktur
folder aplikasi yang sebelumnya menggunakan pendekatan flat, yaitu seluruh file
ditempatkan langsung dalam folder screens/ dan services/, akan diubah menjadi
arsitektur berbasis fitur (feature-based architecture). Pendekatan ini
mengikuti prinsip Separation of Concerns, di mana setiap fitur bisnis
memiliki folder mandiri yang berisi seluruh komponen terkait.

·      
Tujuan kedua adalah implementasi API Service Layer.
Diperlukan lapisan abstraksi yang menangani komunikasi HTTP antara frontend
Flutter dan backend Laravel API. Lapisan ini mencakup konstruksi request,
manajemen token autentikasi ganda (dual token), serta penanganan response
dan error secara terpusat.

·      
Tujuan ketiga adalah integrasi endpoint dan penghapusan
Firebase. Seluruh fitur frontend, mulai dari autentikasi, pencatatan
konsumsi, hingga visualisasi riwayat emisi, akan dihubungkan dengan endpoint
API yang telah dibangun pada Iterasi 1. Setelah integrasi selesai, seluruh
dependensi Firebase SDK akan dihapus dari proyek.

Pada struktur

Pada struktur baru, seluruh file tersebut berada dalam satu folder
features/calculator/ dengan sub-folder masing-masing, sehingga konteks
pekerjaan menjadi lebih terfokus. Berikut hasil reorganisasi
struktur folder secara lengkap:

Gambar IV- 22 hasil reorganisasi struktur folder Frontend

Struktur
di atas terdiri dari tiga lapisan utama. Pertama, folder core/ menyimpan
komponen infrastruktur yang digunakan bersama oleh seluruh fitur, seperti HTTP client
(network/), penyimpanan token (storage/), dan model data umum (models/). Kedua,
folder features/ mengelompokkan kode berdasarkan fitur bisnis. Terdapat empat
fitur, yaitu auth/ untuk autentikasi, calculator/ untuk pencatatan dan
kalkulasi emisi, home/ untuk tampilan beranda dan visualisasi data, serta
profile/ untuk pengelolaan profil pengguna. Setiap fitur memiliki sub-folder
screens/ untuk tampilan, services/ untuk logika akses data, widgets/ untuk
komponen UI yang dapat digunakan ulang, dan models/ untuk representasi data
khusus fitur tersebut. Ketiga, folder shared/, utils/, dan widgets/ menyimpan
utilitas dan komponen yang bersifat generik. Arsitektur ini mendukung
skalabilitas karena penambahan fitur baru cukup dilakukan dengan menambahkan
folder baru di dalam features/ tanpa mengubah struktur yang sudah ada. Sebagai
contoh, jika di masa depan ditambahkan fitur &quot;Tantangan Komunitas&quot;,
cukup dibuat folder features/challenge/ dengan sub-folder screens/, services/,
dan widgets/ di dalamnya.

IV.8.6 Perancangan
API Service Layer

Komunikasi antara frontend
Flutter dan backend Laravel memerlukan lapisan abstraksi yang
disebut API Service Layer. Lapisan ini menangani tiga tanggung jawab
utama, yaitu konstruksi HTTP request, manajemen token autentikasi,
dan penanganan response. Tanpa lapisan ini, setiap screen harus
menulis kode HTTP sendiri yang mengakibatkan duplikasi kode dan inkonsistensi
penanganan error.

Gambar IV- 23 Mengilustrasikan arsitektur lapisan API Service
Layer beserta aliran data dari UI hingga backend

Diagram di atas
menunjukkan lima komponen utama yang tersusun dalam empat lapisan. Pada lapisan
teratas terdapat Flutter UI Layer yang terdiri dari berbagai screen, lapisan
ini tidak mengetahui detail implementasi HTTP dan hanya berinteraksi dengan
service layer. Di bawahnya terdapat service layer yang terdiri dari
AuthServiceAdapter untuk operasi autentikasi dan ConsumptionService
untuk operasi CRUD konsumsi. Kedua service ini menggunakan ApiClient
sebagai komponen terpusat untuk mengirim HTTP request. ApiClient secara
otomatis mengambil token dari TokenStorage dan menyisipkannya ke header
setiap request. Pada lapisan terbawah terdapat Laravel Backend API yang
menerima dan memproses request. Desain ini menerapkan dua prinsip
penting dalam rekayasa perangkat lunak. Pertama, prinsip Dependency Inversion,
di mana komponen UI tidak bergantung langsung pada detail implementasi HTTP,
melainkan bergantung pada abstraksi yang disediakan oleh service layer.
Kedua, prinsip Single Responsibility, di mana setiap komponen memiliki satu
tanggung jawab yang jelas: ApiClient menangani HTTP, TokenStorage menangani
persistensi token, dan masing-masing service menangani logika bisnis spesifik. Khusus
untuk autentikasi, digunakan pola desain Adapter. Pola ini
dipilih karena kode UI lama (yang sebelumnya menggunakan Firebase Auth)
mengharapkan properti currentUser tersedia secara sinkron,
sementara API Laravel bersifat
asinkron. AuthServiceAdapter menjembatani perbedaan ini dengan
menyediakan antarmuka yang kompatibel dengan kode lama, namun secara internal
memanggil API Laravel. Dengan pendekatan ini, perpindahan dari Firebase Auth ke
API Laravel tidak memerlukan perubahan menyeluruh pada seluruh screen yang
menggunakan autentikasi.
